# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page Streamlit prototype ("Neural Taste Profile") built for **RFP 2 (Zomato)** of a Searce internal intern assignment on AI & Digital Transformation, with a Profile modal dialog for one-time dietary/health/biometric setup. It's the technical proof-of-concept for one claim: food can be represented as a multi-dimensional **flavor vector** instead of a cuisine label, real-time context (weather, biometrics, mood, craving) can be fused into a target flavor vector, and cosine similarity over those vectors — restricted to dishes that are actually safe for the user to eat — produces better, explainable recommendations than category browsing.

**The context-fusion step is LLM-driven, not rule-based.** Earlier versions of this prototype used hardcoded delta tables for every input signal (time/mood/activity/keyword). That approach has been replaced: `llm_context_engine.py` sends the full real-time context (time of day, weather, biometrics, mood-quiz answers, free-text craving) to the Claude API in a single call and gets back the 14-dimension target flavor vector **plus a per-dimension rationale**, so the "why this matched" explanation stays legible instead of becoming a black box. This is the one place in the app that calls an external LLM — everything downstream (`dietary_filter.py`, `matching_engine.py`) remains plain deterministic code, and a deterministic fallback exists for when the API call fails (see below). Don't reintroduce a parallel rule-based context-fusion path outside this scope; extend `llm_context_engine.py`'s prompt/schema instead.

**Dietary/allergy safety constraints are a deterministic pre-filter, not part of the LLM step or the similarity score.** A user's dietary preference (vegetarian/vegan), allergies, and celiac/lactose-intolerance status are hard rules — `dietary_filter.apply_hard_filters()` removes any violating dish from the candidate list *before* `matching_engine.rank_dishes()` ever sees it, so no similarity score can resurrect an unsafe dish. Risk-based health conditions (diabetic/high cholesterol/hypertension) are softer — `dietary_filter.apply_soft_penalties()` down-weights an already-computed score within the safe set instead of excluding anything. Both live in `dietary_filter.py`, never in the LLM payload. See that module's entry below.

## Commands

```bash
cd app
python3 -m venv .venv && .venv/bin/pip install -r requirements.txt
cp .env.example .env   # then fill in ANTHROPIC_API_KEY
.venv/bin/streamlit run app.py   # http://localhost:8501
```

Without `ANTHROPIC_API_KEY` set, the app still runs — it just always shows the fallback banner and uses the deterministic baseline vector instead of the LLM-authored one. `app.py` calls `load_dotenv()` at startup, which reads `app/.env` (gitignored; `app/.env.example` is the committed template).

Tests (`app/tests/`):
```bash
cd app
.venv/bin/pip install -r requirements-dev.txt   # adds pytest on top of requirements.txt
.venv/bin/pytest                 # runs the whole suite; testpaths is set in pytest.ini
.venv/bin/pytest tests/test_matching_engine.py -v   # a single file
.venv/bin/pytest -k explain_match                   # by name
```

The test suite makes **zero** real network or API calls — every Claude API call and every Open-Meteo call is monkeypatched (see "Tests" below). It runs the same with or without `ANTHROPIC_API_KEY` set.

A `Makefile` in `app/` wraps the above: `make install-dev` (venv + all deps), `make run`, `make test` / `make test-verbose`, `make clean`.

There is no linter or backend — this is a static prototype with one external network dependency (the Claude API call) and one optional one (Open-Meteo weather), no auth, no persistence, no server of its own.

## Architecture (`app/`)

Dataset, engines, charts, and UI are split across flat modules (no packages/`__init__.py` — Streamlit adds the script's own directory to `sys.path`, so plain top-level imports like `from dishes import DISHES` resolve). Each module is a self-contained stage of the pipeline; `app.py` only wires them to widgets and holds no data or matching logic itself.

1. **`dishes.py`** — the single source of truth for vector shape. `DIMENSIONS` (the canonical list of 14 flavor dimensions: spice, acidity, richness, warmth, crunch, umami, sweetness, bitterness, saltiness, freshness, moisture, aroma, chewiness, temp_contrast — order matters since `vec(...)` builds dish vectors positionally from it), `DIM_LABELS` (display names), and `DISHES` (400 hand-authored dishes across 16 cuisine buckets, each with a 14-dim `vector` plus `cuisine`/`prep_time_minutes`/`meal_type` metadata, plus the dietary/health fields `dietary_filter.py` reads: `is_vegetarian`/`is_vegan`/`is_gluten_free`/`is_dairy_free`/`allergens` (hand-authored real-world facts — none of this is derivable from the flavor vector) and `sugar_level`/`fat_level`/`sodium_level` (0-10, same convention as the flavor vector, seeded from a heuristic correlated with sweetness/richness+crunch/saltiness+umami with manual overrides where that heuristic clearly under/over-shoots — see the module docstring). The `diet(...)` helper builds this metadata dict positionally, mirroring `vec(...)`'s style. Every other module imports `DIMENSIONS`/`DIM_LABELS` from here.
2. **`context_engine.py`** — now a small module: `get_time_bucket(now)` buckets a datetime's hour into Morning/Afternoon/Evening/Late Night; `TIME_DELTAS` is a fixed delta table (kept from the old rule-based design) that does double duty as a labeled input sent to the LLM and as the backbone of the deterministic fallback vector; `QUIZ_QUESTIONS` is the mood-quiz question bank (a handful of optional multiple-choice questions, replacing the old single mood chip + activity multiselect); `add_delta`/`clamp_vector` are small vector utilities shared with `llm_context_engine.py`'s fallback path.
3. **`weather_client.py`** — calls Open-Meteo's free current-weather endpoint for a lat/lon. `PRESET_CITIES` is the dropdown of major Indian cities (plus a "Custom location" option). `get_current_weather(lat, lon)` returns a dict on success or `None` on **any** failure (network error, timeout, unexpected response shape) — it never raises. A weather-fetch failure is a small, separate failure surface from the LLM fallback below: `app.py` just omits weather from the context payload and shows a small warning.
4. **`geolocation_client.py`** — reads the browser's real GPS location via `streamlit_js_eval`'s wrapper around the JS `navigator.geolocation` API. `get_browser_location()` returns `{"lat", "lon"}` on success or `None` on **any** failure (permission denied, unsupported browser, or not yet responded) — same never-raise contract as `weather_client.get_current_weather`. `app.py` calls this unconditionally on every page load (no opt-in gate): the browser's native permission prompt fires once per origin and is remembered by the browser on subsequent reruns. On success it overrides whatever lat/lon the manual city dropdown computed, and triggers exactly one automatic `weather_client.get_current_weather` call per detected coordinate (guarded by `st.session_state.auto_weather_location` so an unrelated widget interaction — which reruns the whole script — doesn't refetch weather every time). The dropdown + "Custom location" lat/lon inputs and the manual "Check Weather" button remain untouched as a fallback for when geolocation is denied/unavailable.
5. **`llm_context_engine.py`** — the Context Engine, and the only module that calls an external LLM. `build_context_payload(...)` assembles the structured dict (time bucket, weather-or-`None`, biometrics-or-`None`, quiz answers, craving text) sent as the user-turn content. `FlavorVectorResponse` (a Pydantic model, one `DimensionScore(value, rationale)` field per dimension in `DIMENSIONS`) is passed as `output_format` to `client.messages.parse(...)` (model: `claude-sonnet-5`) — Sonnet 5 runs adaptive thinking on by default, which is why no extra `thinking` config is needed for "decent reasoning depth." `DimensionScore`'s `value` field clamps to `[0, 10]` via a `field_validator` rather than rejecting the whole response over one out-of-range number (the API's JSON schema strips numeric range constraints before it reaches the model, so the 0-10 instruction lives in the system prompt as text, backed by this client-side clamp). `get_target_vector(context_payload)` wraps the whole call in a broad `try/except` — any failure (network, rate limit, refusal, malformed response, timeout) routes to `_fallback_vector_and_rationale(...)`, a small deterministic computation (`{dim: 5} + TIME_DELTAS[bucket]` plus a handful of fallback-only quiz-answer and craving-keyword deltas, `FALLBACK_QUIZ_DELTAS`/`FALLBACK_KEYWORD_DELTAS` — deliberately much smaller than the old rule-based tables, kept only so the fallback isn't flatly identical for every user at the same hour). Always returns a `ContextResult(vector, rationale, source, error)` — never raises. `_get_client()` is a small seam (constructs `anthropic.Anthropic()` lazily, not at import time) that tests monkeypatch directly.
6. **`dietary_filter.py`** — the deterministic safety layer between `llm_context_engine` and `matching_engine`, no LLM, no Streamlit. `ALLERGEN_OPTIONS` and `HEALTH_CONDITIONS` are the canonical vocabularies shared with the Profile dialog's widgets and every dish's `allergens`/health-relevant fields, so the two can't drift apart. `apply_hard_filters(dishes, profile)` is the pre-filter: it returns `(allowed, exclusion_report)`, where `allowed` is the subset of dishes violating none of the user's hard constraints (dietary preference, celiac→`is_gluten_free`, lactose-intolerant→`is_dairy_free`, any stored allergy against a dish's `allergens`) and `exclusion_report` carries `total_excluded`/`by_reason`/`excluded` for the UI's transparency message. This must run *before* `rank_dishes` — `app.py` always calls it first — so a disallowed dish is never a similarity candidate, and no score can override the rule. `apply_soft_penalties(ranked, profile)` runs *after* `rank_dishes`, only within the already-filtered set: it down-weights a dish's score in proportion to its `sugar_level`/`fat_level`/`sodium_level` for whichever of diabetic/high_cholesterol/hypertension are active, re-sorts, and never excludes anything (celiac/lactose_intolerant are ignored here — they're fully handled upstream as hard constraints). The hard/soft split lives in this module's code, not in the Profile dialog's shared "Health conditions" checkbox grouping.
7. **`matching_engine.py`** — the Matching Engine: `cosine_similarity(vec_a, vec_b)` (numpy) and `rank_dishes(target_vector, dishes)` do the actual matching — both unchanged by the LLM pivot and by the dietary-filter addition, fully deterministic; `explain_match(target_vector, dish, rationale)` finds the top 2-3 dimensions by `target*dish` product (still deterministic and instant, no LLM call at explain-time) and pulls the "why this matched" text from `rationale[dim]` — the per-dimension strings authored by the LLM, or by the deterministic fallback when the LLM call failed.
8. **`charts.py`** — `make_radar_chart(vector, title)` builds a Plotly `Scatterpolar` figure from a 14-dim vector dict. Unaffected by the LLM pivot.
9. **`app.py`** — Streamlit UI only, single-page script (no `st.Page`/`st.navigation` — that was tried and reverted; see `profile_dialog.py` below for why). Right under the title/caption: a profile-summary caption (`profile_dialog.profile_summary()`) plus an "Edit Profile" button that calls `profile_dialog.edit_profile_dialog()` on click. Then: auto-detected time bucket → location (automatic browser geolocation attempt on every load via `geolocation_client.get_browser_location()`, which also auto-triggers a weather fetch on success; the city dropdown/"Custom location" lat-lon inputs + manual "Check Weather" button remain as a fallback for when geolocation is denied/unavailable, fetched *before* "Find My Meal" so the network calls don't stack into one wait) → biometrics read from the Profile dialog's session_state (see `profile_dialog.py` below; a caption points there if no smartwatch is connected) → mood quiz (`QUIZ_QUESTIONS`, each optional) → free-text craving → "Find My Meal" (wrapped in `st.spinner`). On click: `llm_context_engine.get_target_vector(...)` produces the target vector (unchanged inputs/outputs), then `dietary_filter.apply_hard_filters(DISHES, profile)` pre-filters *before* `rank_dishes(result.vector, allowed)` runs, then `dietary_filter.apply_soft_penalties(...)` re-scores within that safe set, then the top-5 slice. Results (including whether the fallback path fired, the raw error for a debug expander, and the `exclusion_report` for the "Excluded N dish(es)..." caption) are stored in `st.session_state.results` on click, so they persist across unrelated reruns (e.g. typing in the craving box) instead of disappearing. Calls `load_dotenv()` at startup, and calls `llm_context_engine.get_target_vector(...)`, `weather_client.get_current_weather(...)`, and `geolocation_client.get_browser_location(...)` via qualified module access (`import llm_context_engine`, not `from llm_context_engine import get_target_vector`) — this is the seam that lets tests monkeypatch each function directly.
10. **`profile_dialog.py`** — the single place a user sets dietary preference (radio: Non-vegetarian/Vegetarian/Vegan), allergies (multiselect from `dietary_filter.ALLERGEN_OPTIONS`), health conditions (checkboxes from `dietary_filter.HEALTH_CONDITIONS`), and the (simulated) smartwatch/biometrics sliders, instead of re-asking every visit. `edit_profile_dialog()` is decorated `@st.dialog("Edit Profile", width="large")` and opened from `app.py`'s "Edit Profile" button; `profile_summary()` builds the one-line caption shown next to that button. Session-level only (`st.session_state`), no database — explicitly out of scope for this prototype. **Note on an earlier design that was tried and reverted**: an `st.Page`/`st.navigation` 2-page version of this app page-scoped widget-keyed session_state (a value living only under a widget's `key=` was dropped the instant you navigated to another page — confirmed empirically), which required a two-key shadow-copy workaround. A dialog is not a page — `ctx.page_script_hash` never changes when it opens/closes — so that problem doesn't apply here: every field uses a single, plain widget `key=` (`"dietary_preference"`, `"allergies"`, `"health_conditions"`, `"smartwatch_connected"`, `"hrv"`, `"calories_burned"`, `"sleep_score"`) that `app.py` reads directly, no shadow key needed. `health_conditions` is still a derived list built from N individual `health_{code}` checkboxes — that aggregation is a data-shape requirement of `dietary_filter.py`, unrelated to any page/dialog-scoping question.

`dishes.py`, `context_engine.py`, `weather_client.py`, `geolocation_client.py`, `dietary_filter.py`, `llm_context_engine.py`, and `matching_engine.py` are all plain, side-effect-free-at-import functions/data with no Streamlit calls — that's deliberate, so this logic can be imported and tested independently of the running app (and, for `weather_client`/`geolocation_client`/`llm_context_engine`, without ever making a real network/API call in tests).

### Adding a dish
Append to `DISHES` in `dishes.py` using `vec(...)` with 14 positional values in the exact `DIMENSIONS` order, plus `diet(...)` for the 8 dietary/health fields. Scores should stay plausible relative to existing dishes in the same cuisine bucket. The boolean/allergen fields must be hand-reasoned from what the dish actually is (not derived from the flavor vector); sugar/fat/sodium levels can start from the correlated-heuristic described in the module docstring but should get a manual override if that heuristic clearly misses (e.g. a dessert with a moderate `sweetness` flavor score should still get a high `sugar_level`).

### Adding a quiz question
Add an entry to `QUIZ_QUESTIONS` in `context_engine.py` (`key`, `question`, `options`). Flavor deltas are no longer hand-authored for the primary path — the LLM interprets quiz answers directly as part of the context payload. If you want the new question's answers to meaningfully affect the *fallback* path too, add corresponding entries to `FALLBACK_QUIZ_DELTAS` in `llm_context_engine.py` (keep it small and explicit, matching the existing entries' style).

### Verifying logic changes
Prefer running the test suite (`pytest`) over ad-hoc scripts. `tests/` imports `dishes`/`context_engine`/`weather_client`/`geolocation_client`/`dietary_filter`/`llm_context_engine`/`matching_engine` directly (no `streamlit` stub needed — none of those seven modules import `streamlit`) and covers dataset shape, `TIME_DELTAS`/quiz-question shape, weather-client success/failure (mocked `requests.get`), geolocation-client success/denial/pending (mocked `streamlit_js_eval.get_geolocation`), dietary-filter hard-filter/soft-penalty correctness, the LLM context engine's happy path/fallback triggers/clamping (mocked Anthropic client via `_get_client`), and matching/explanation correctness. Add a test alongside the existing ones in the relevant test file rather than writing a throwaway script.

Run `.venv/bin/pip install -r requirements.txt` after editing `requirements.txt`; it now includes `streamlit`, `numpy`, `plotly`, `anthropic`, `requests`, `pydantic`, `python-dotenv`, `streamlit-js-eval`. `requirements-dev.txt` layers `pytest` on top for the test suite — don't add test-only deps to `requirements.txt`. No new dependency was needed for the Profile dialog — `st.dialog` ships with the already-installed Streamlit version.

## Tests (`app/tests/`)

`conftest.py` puts `app/` on `sys.path` so test files import the app modules as plain top-level modules (`from dishes import DISHES`) regardless of the directory `pytest` is invoked from.

- **`test_dishes.py`** — dataset shape: dish count, unique ids/names, all 14 dims present and in 0-10, required metadata, `vec()` positional mapping, and the 8 dietary/health fields (correct types/ranges across all 400 dishes, the `is_vegan → is_vegetarian` invariant, and that `is_gluten_free`/`is_dairy_free` never contradict the `allergens` list).
- **`test_context_engine.py`** — `TIME_DELTAS` bucket coverage and dimension validity, `get_time_bucket` boundary conditions (parametrized across all 4 bucket edges), and `QUIZ_QUESTIONS` shape (unique keys, non-empty questions, ≥2 options each).
- **`test_weather_client.py`** — `get_current_weather` mocked against `requests.get`: success (parses the Open-Meteo payload correctly), and returns `None` (never raises) on a connection error, a bad HTTP status, or a malformed payload. No real network calls.
- **`test_geolocation_client.py`** — `get_browser_location` mocked against `streamlit_js_eval.get_geolocation`: success (parses the `coords` dict into `{"lat","lon"}`), and returns `None` (never raises) on a permission-denied error dict, a still-pending `None` result, or a malformed payload. No real browser/JS execution.
- **`test_dietary_filter.py`** — `apply_hard_filters`: no-restriction returns every dish; each hard constraint (vegetarian/vegan/celiac→gluten-free/lactose→dairy-free/allergy) excludes exactly the right dishes with correct `by_reason` counts; a dish violating two constraints is counted under both reasons; and a directly-encoded core-guarantee test that hard-filtered-then-`rank_dishes`-fed output never contains an excluded dish id. `apply_soft_penalties`: score lowers proportional to level, is a no-op with no active conditions, never changes dish membership/count, output stays sorted best-first, and celiac/lactose_intolerant produce zero penalty even if passed in (already fully handled upstream as hard constraints).
- **`test_llm_context_engine.py`** — `DimensionScore`'s clamping validator (above/below range), `get_target_vector`'s happy path, refusal, and exception paths (all via monkeypatching `_get_client()` to return a fake client — no real Anthropic call), and the deterministic fallback formula (baseline + `TIME_DELTAS` + fallback-only quiz/keyword deltas, and that it stays within 0-10 and actually varies with input).
- **`test_matching_engine.py`** — `cosine_similarity` properties (self=1, symmetric, no div-by-zero on a zero vector), `rank_dishes` ordering, and `explain_match` correctness against hand-built `rationale` dicts (top-dimension selection, the 2-reason cap for readability, graceful handling of a dimension with no rationale entry).
- **`test_charts.py`** — `make_radar_chart` returns a `plotly.graph_objects.Figure`, closes the polygon (first point repeated at the end), and handles every dish in the dataset.
- **`test_app.py`** — integration tests that run the *actual* `app.py` headlessly via Streamlit's official `streamlit.testing.v1.AppTest` harness (no browser, key-based widget lookups like `at.button(key="find_meal_btn")` rather than positional indices, since the widget list has grown). Dialogs render into the root element tree's `event` block, which `AppTest`'s `at.radio(key=...)`/etc. accessors already traverse — no special dialog API needed, tests just click `at.button(key="edit_profile_btn").click().run()` to open it and then interact with its widgets by key like any other. The one harness quirk: `AppTest.run()` has no fragment-scoped rerun (real Streamlit keeps a dialog open across a widget interaction inside it via a fragment-only rerun; `AppTest` always re-runs the whole script), so a test that needs to change a value inside an already-open dialog on a *later* run must re-click `edit_profile_btn` on that same run too — or, simpler and just as legitimate, seed the target value directly into `at.session_state[...]` before the dialog's first render (the documented way to preset a widget's initial value) and skip simulating the click on that field entirely, which is what most of these tests do. None of these tests make a real Claude API, weather, or geolocation call: `llm_context_engine.get_target_vector` is monkeypatched (module-attribute style — `app.py` calls it via `import llm_context_engine` specifically so this works) to either a fake result (for the mocked-happy-path and "different inputs" tests) or, for the one test that needs an exact numeric expectation, `_get_client` is monkeypatched to raise so the *real* fallback code path runs and can be asserted against an independently-computed expectation — mirroring how `test_llm_context_engine.py` verifies the same formula directly. `geolocation_client.get_browser_location` and `weather_client.get_current_weather` are monkeypatched the same way to cover: geolocation success auto-fetching weather exactly once on load, geolocation failure/denial falling back to the manual dropdown, and the detected location + weather surviving an unrelated rerun without a redundant weather refetch (real `AppTest` runs never execute real browser JS, so the unpatched component always resolves to `None` there — safe, since that's the same "not yet responded" case the seam already handles). Also covers: the fallback warning banner (and its debug expander) appearing when forced, session-state persistence across an unrelated rerun, a vegetarian dietary preference set via the Profile dialog hard-excluding a non-veg dish from results even when its vector is a perfect similarity match (and the exclusion caption appearing), smartwatch biometrics set via the Profile dialog flowing into the LLM payload, the profile-never-visited defaults path still working normally, and the dialog itself opening and rendering its widgets with the expected current values.

When adding a dish or a quiz question (see above), add or extend a test in the relevant test file rather than only manually re-running the app.
